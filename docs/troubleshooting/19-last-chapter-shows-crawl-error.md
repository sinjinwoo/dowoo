# 마지막 화에서 "다음 편" 클릭 시 "다음 편이 없습니다" 대신 크롤링 오류가 뜸

## 증상

69shuba.com 소설의 마지막 화에서 "다음 편"을 누르면, 정상적인 안내("다음 편이 없습니다.") 대신 아래와 같은 실제 크롤링 실패 응답이 그대로 에러 모달로 노출됐다.

```json
{
    "status": 502,
    "data": null,
    "message": "본문/제목을 추출하지 못했습니다(사이트 마크업 변경 추정).",
    "error": { "code": "PARSE_FAILED", "details": null }
}
```

## 원인

두 가지가 겹쳐서 발생했다.

**1) `.htm` / `.html` 확장자 혼동**

69shuba.com은 첫 장/마지막 장에서 이전·다음 링크가 실제 회차가 아니라 목차 페이지 `book/{bookId}.htm`(확장자에 `l`이 없다)을 가리킨다. 이 프로젝트에서 "마지막 화 여부"를 판별하는 `NO_CHAPTER_RE` 정규식을 처음엔 `.htm`으로 만들었다가, 도중에 `.html`로 잘못 정정했다가, 실제로는 다시 `.htm`이 맞는 것으로 확인되는 등 이 한 글자를 두고 여러 차례 오락가락했다. 정규식이 실제 사이트 값과 다르면 "마지막 화 도착" 신호(목차 페이지 링크)를 회차 링크로 착각해서 그 URL을 그대로 크롤링 시도하고, 목차 페이지에는 `.txtnav`(본문)도 `h1.hide720`(제목)도 없으니 `PARSE_FAILED`로 이어졌다.

**2) 이미 저장된 챕터의 stale `nextUrl`**

`nextUrl`은 최초 크롤링 시점에 계산되어 DB에 저장되고, 이후에는 재검증해도 DB 값 자체가 갱신되지 않는다. 정규식을 고치기 전에 이미 크롤링해둔 챕터는 버그 있던 파서가 계산한 잘못된(= null이어야 하는데 null이 아닌) `nextUrl`을 그대로 갖고 있어서, 프론트의 재검증 로직(`activeChapter.nextUrl`이 `null`일 때만 재크롤링해서 재확인)이 아예 발동하지 않고 곧장 그 stale한 주소로 크롤링을 시도했다.

**3) (부가) 재검증으로 얻은 주소도 100% 신뢰할 수는 없음**

`App.tsx`의 "재검증" 로직은 현재 챕터를 다시 크롤링해서 `nextUrl`이 새로 생겼는지 확인하는데, 이렇게 얻은 주소는 아직 실제로 크롤링해본 게 아닌 추정값이다. 사이트에 따라 다음 회차 URL을 미리 채번해두고 실제 콘텐츠는 없는 경우가 있어, 이 추정 주소를 크롤링했을 때 실패하더라도 "크롤링 오류"로 보여주면 사용자가 오해하기 쉽다.

## 해결

- `dowoo-python/app/crawl/parsers/shuba69.py`: `NO_CHAPTER_RE`가 `.htm`/`.html` 둘 다 매치하도록 `re.compile(r"/book/\d+\.html?$")`로 변경. 이 한 글자를 두고 반복적으로 헷갈렸던 지점이라, 어느 쪽이 맞는지 매번 재확인하는 대신 정규식 자체를 관대하게 만들어 근본적으로 방지했다.
- `dowoo/src/App.tsx`의 `handleNavigateChapter`: 재검증으로 얻은 `nextUrl`을 `isSpeculativeNext` 플래그로 표시해두고, 이 주소로 실제 크롤링을 시도했다가 실패하면 `type: 'crawling'`(크롤링 오류)이 아니라 `type: 'boundary'`(다음 편이 없습니다)로 처리하도록 분기했다. 이미 DB에 저장돼 known-good인 `nextUrl`로 이동하다 실패한 경우는 여전히 실제 크롤링 오류로 표시된다.
- 이미 저장된 stale한 챕터 데이터는 코드 수정만으로는 갱신되지 않으므로, 검증 시에는 문제였던 소설/챕터를 삭제 후 재크롤링해서 새 로직으로 다시 저장된 값을 확인해야 한다.

## 참고

- 사이트 마크업 문자열(특히 확장자처럼 한 글자 차이로 완전히 다른 뜻이 되는 값)을 사람이 대화로 여러 번 주고받으며 확인하면 오탈자가 생기기 쉽다. 가능하면 정규식을 관대하게 만들어(`.html?`처럼) 어느 쪽이든 매치하게 하는 편이 반복 확인보다 안전하다.
- 크롤링 결과로 얻은 `nextUrl`/`prevUrl`은 "크롤링 시점의 스냅샷"이라는 걸 기억할 것 - 파서 로직을 나중에 고쳐도 이미 저장된 챕터의 값은 자동으로 갱신되지 않는다. 재크롤링 없이는 낡은 값이 계속 쓰인다.
- "다음 편이 존재하는지"를 추정하기 위해 수행한 크롤링(재검증)과, 사용자가 실제로 보려는 콘텐츠를 가져오기 위한 크롤링은 신뢰 수준이 다르다 - 전자에서 비롯된 실패는 사용자에게 원인 불명의 에러로 보여주기보다 "경계 도달"로 흡수하는 편이 UX상 낫다.
- 관련 파일: `dowoo-python/app/crawl/parsers/shuba69.py`, `dowoo/src/App.tsx`
