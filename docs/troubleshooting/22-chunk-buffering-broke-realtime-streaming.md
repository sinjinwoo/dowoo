# 청크 번역 도입 후 실시간 스트리밍이 안 되고 번역이 끝나야 한 번에 보임

## 증상

청크 번역(1만자 단위 분할) 기능을 넣은 뒤, 번역문이 줄 단위로 실시간으로 나타나던 것이 사라지고
번역이 전부 끝난 다음에야 화면에 한꺼번에 표시됐다.

## 원인

청크 단위로 모델/키를 재시도할 때 "실패한 시도의 부분 출력이 다음 재시도 결과와 섞이면 안 된다"는
안전장치를 넣으면서, `_try_one_key`가 스트리밍 중 받은 줄을 즉시 내보내지 않고 청크 하나가 완전히
끝날 때까지 내부 리스트에 모아뒀다가 `_chunk_done` 이벤트로 한 번에 넘기도록 짜버렸다.

```python
# 문제가 됐던 구조 - 청크가 끝나야만 lines를 한꺼번에 반환
async for chunk in stream:
    ...
    translated_lines.append(line)   # 여기서 바로 안 내보내고 모으기만 함

if translated_lines:
    yield {"event": "_chunk_done", "data": {"lines": translated_lines}}  # 다 모은 뒤에야 전달
```

청크 크기가 1만자라 대부분의 챕터(1만자 이하)는 청크가 하나뿐이었고, 그 결과 챕터 전체가 번역
완료될 때까지 아무것도 화면에 나타나지 않다가 끝나는 순간 전부 한꺼번에 쏟아지는 것처럼 보였다.
"실패 시 부분 출력 격리"라는 목표 자체는 맞았지만, 그 수단으로 실시간 스트리밍이라는 핵심 기능을
희생시킨 것이 문제였다.

## 해결

`_try_one_key`가 줄이 완성되는 즉시 `_line` 이벤트로 내보내도록 되돌렸다. 대신 청크 단위
재시도(모델/키 순환)나 RPM 분산을 위한 라운드로빈 로직 자체는 그대로 유지했다 - 청크 "경계"는
여전히 1만자 단위지만, 청크 "내부"에서는 줄 단위로 즉시 스트리밍한다.

```python
async for chunk in stream:
    ...
    translated_lines.append(line)
    yield {"event": "_line", "data": line}   # 완성되는 즉시 내보냄
```

"시도가 중간에 실패하면 이미 내보낸 부분 출력이 재시도 결과와 섞일 수 있다"는 위험 자체는 남지만,
이건 청크 번역을 도입하기 전(챕터 전체를 한 번에 요청하던 시절)에도 이미 있던 것과 동일한
수준의 위험이다 - 이 프로젝트에서 실제로 문제가 된 적은 없었고, 실시간 스트리밍이라는 핵심 기능을
잃는 것보다는 감수할 만한 트레이드오프로 판단했다.

## 참고

- 재시도/에러 격리 같은 안전장치를 넣을 때, 그 안전장치가 스트리밍의 "실시간성" 자체를 깨뜨리지
  않는지 반드시 확인할 것. "끝난 뒤 한꺼번에 검증하고 내보내기"는 안전해 보이지만 스트리밍
  아키텍처에서는 사실상 스트리밍을 꺼버리는 것과 같다.
- 리팩터링 시 기존 기능(여기서는 "줄 단위 실시간 표시")이 암묵적으로 어떤 코드에 의존하고
  있었는지 목록화해두지 않으면, 다른 목적(청크 재시도 안전성)으로 코드를 고치다가 조용히
  회귀시키기 쉽다. 사용자가 실제로 써보고 나서야 발견된 문제였다.
- 관련 파일: `dowoo-python/app/translate/gemini_client.py`
