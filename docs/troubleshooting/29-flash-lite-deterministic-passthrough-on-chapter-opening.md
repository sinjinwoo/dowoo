# 특정 화만 어떤 키/모델로도 계속 미번역(원문 그대로) 판정에 걸림

## 증상

[23번 문서](23-gemini-flash-lite-untranslated-passthrough.md)의 미번역 감지·재시도 로직이
있는데도, 특정 화 하나만 가진 API 키를 다 돌려도 계속 실패했다("번역 루프에 걸림"). 다른
화는 문제없이 잘 번역됐다.

## 원인

임시 로그(`[TEMP DEBUG]`, 이번 진단 후 제거)로 실제 모델 응답을 찍어보니, 매 시도(서로 다른
키 포함)마다 **똑같이** 챕터 제목 줄과 그 다음 문장을 번역하지 않고 원문(중국어) 그대로
반환하고 있었다. `pending_cjk_count`도 20자 문턱치를 이미 넘긴 상태였고 `_looks_untranslated`도
정확히 동작하고 있었다 - 즉 코드 버그가 아니라, **모델(gemini-3.1-flash-lite)이 이 청크에서
결정론적으로 앞부분을 베끼는 결함**이었다.

이 화는 챕터 제목 형식이 독특하고(「」 특수 괄호), 본문 뒤쪽에 【任務...】식 브라켓 시스템
메시지 블록과 사이트 광고 워터마크(옵스큐어 유니코드 결합 문자로 도메인을 가린 문구, 예:
`t̸w̸k̸a̸n̸.c̸o̸m̸`)까지 섞여 있어 구조가 다른 화보다 유별났다. 이런 구조적 이질성이 가벼운
모델의 "이 부분은 그대로 둬야 하나?" 오판을 유발한 것으로 보인다. 재시도가 사실상 무의미했던
이유도 여기 있다 - 실패가 샘플링 변동성이 아니라 그 청크 내용 자체의 특성이라, 키를 아무리
바꿔도(모델을 안 바꾸는 한) 재현됐다.

추가로, 당시 기본 시스템 프롬프트(`DefaultPrompts.java`)에는 이 앱에서 실제로는 전혀 쓰이지
않는 지침("입력에 이미 번역된 앞부분이 함께 주어질 수도 있다")이 섞여 있어, 모델이 청크
맨 앞부분을 "이미 처리된 부분이니 건드리지 말아야 하나"로 오판할 여지를 더 키웠을 가능성이
있다.

## 해결

`dowoo-back/src/main/java/io/dedyn/jwlabs/dowoo/library/support/DefaultPrompts.java`의
기본 시스템 프롬프트를 아래 방향으로 다시 썼다.

1. 실제로 쓰이지 않는 "이미 번역된 앞부분" 지침 삭제.
2. 이 프로젝트는 중국어 소설만 지원하므로 일본어 관련 지침(고유명사 표기 규칙 등) 전부 삭제.
3. "최우선 명령"으로 격상해 더 명시적으로 못박음 - 본문 전체가 중국어로 나오거나 미번역
   중국어 단어가 단독으로 남으면 규칙 위반, 고유명사/무공명/도가·불교 용어에 한해서만
   한글 뒤 한자 병기 허용, 그 외 일반 서술어·조사는 절대 원문 그대로 복사 금지.

실사용 검증 결과, 이 프롬프트로도 100% 즉시 성공하진 않았지만(여전히 몇 번의 재시도가 필요할
때가 있음) 결국 재시도 몇 번 안에 성공하는 수준까지 실패율이 낮아졌다 - 완전히 결정론적인
결함이 아니라 프롬프트로 확률을 낮출 수 있는 문제였다는 뜻이다.

**주의**: `DefaultPrompts.SYSTEM_PROMPT`는 신규 소설 생성 시점에만 DB(`NovelPrompt.systemPrompt`)로
복사된다(`TranslateService.java`, `NovelService.java`, `ReadService.java`). 이미 생성된
소설은 이 변경이 자동 반영되지 않으므로, 기존 소설에 적용하려면 앱에서 해당 소설의 시스템
프롬프트를 직접 다시 저장해야 한다.

## 참고

- 결정론적으로 재현되는 실패(같은 입력 → 항상 같은 결과)는 재시도/키 로테이션으로 못 고친다 -
  로그로 "매번 똑같이 실패하는지" 먼저 확인하고, 그렇다면 재시도 전략이 아니라 입력(프롬프트,
  콘텐츠)을 의심할 것.
- 프롬프트에 죽은 지침(실제로 발동 조건이 없는 문장)이 섞여 있으면 모델이 불필요한 모호성을
  갖게 될 수 있다 - 주기적으로 "이 지침이 지금도 실제로 쓰이는 시나리오인가"를 점검할 것.
- 강한 프롬프트도 완전한 보증은 아니다 - 특히 경량 모델(flash-lite)은 지침 준수력이 낮아
  프롬프트를 더 길고 복잡하게 만들면 오히려 다른 부작용이 생길 수 있다. 프롬프트 개선은
  "실패 확률을 낮추는 것"으로 기대치를 잡고, 근본 대책은 더 강한 모델을 폴백으로 두는 것.
- 관련 파일: `dowoo-back/src/main/java/io/dedyn/jwlabs/dowoo/library/support/DefaultPrompts.java`
