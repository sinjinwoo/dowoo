# 긴 챕터를 청크로 나눠 번역할 때, 청크마다 같은 키만 계속 쓰면 RPM 한도에 몰릴 위험

## 증상

없음(실제 장애 아님) - 청크 번역 기능을 설계하는 단계에서 리뷰 중 발견한 문제. "청크마다 모델/키
순환을 처음부터 다시 하지 않고, 직전 청크에서 성공한 조합을 그대로 이어서 쓴다"는 최적화를
그대로 두면, 청크가 많은(원문이 아주 긴) 챕터에서 실제로 RPM(분당 요청 수) 한도를 건드릴 수
있는 설계였다.

## 원인

`gemini_client.py`의 `translate_stream()`은 원문을 1만자 단위 청크로 나눠 순차 번역하는데,
청크마다 매번 무작위로 모델/키를 다시 고르면 이미 잘 되는 조합이 있는데도 실패를 반복할 수
있어 "직전 청크에서 성공한 (모델, 키) 조합을 다음 청크에서도 그대로 사용"하도록 만들었다.

```python
# 최초 구현 - 성공한 키를 그대로 다음 청크에도 사용
model_start_index = model_index
key_start_index = key_index
```

문제는 이 방식이 "청크가 여러 개인 긴 챕터 하나"의 모든 청크 요청을 **같은 키 하나**에 몰아준다는
점이다. API 키를 여러 개 등록해서 로테이션하게 만든 이유 자체가 "한도 분산"인데, 청크가 3~5개
이상인 긴 챕터에서는 그 챕터 하나를 처리하는 동안 등록된 다른 키들은 전혀 쓰이지 않고 첫 번째로
성공한 키에만 짧은 시간 안에 요청이 연달아 몰린다 - 등록된 키가 여러 개라도 RPM 한도 앞에서는
사실상 키가 하나뿐인 것과 비슷한 상황이 된다.

## 해결

성공한 **모델**은 그대로 유지하되(같은 챕터 안에서 청크마다 다른 모델을 쓰면 번역 톤이 달라질 수
있어서), **키**는 청크가 성공할 때마다 다음 키로 라운드로빈하도록 바꿨다.

```python
model_start_index = model_index
# 성공한 키를 그대로 또 쓰지 않고 다음 키로 넘긴다 - 안 그러면 청크가 많은 긴 챕터에서
# 계속 같은 키에만 요청이 몰려 RPM 한도에 걸리기 쉽다. RPM 쿼터는 보통 키+모델 조합별이라
# 모델을 유지해도 키만 바꾸면 그 모델의 부하가 여러 키로 분산된다.
key_start_index = (key_index + 1) % len(keys)
```

이러면 청크1이 key-a로 성공하면 청크2는 key-b부터 시도하고, 청크3은 다시 key-a부터... 식으로
등록된 모든 키에 부하가 고르게 분산된다. 특정 키가 그 사이 무효화/한도 초과가 됐다면 기존 재시도
로직이 그대로 작동해 다음 키로 자동 폴백한다.

## 참고

- 여러 키를 순환하는 로직을 만들 때, "직전에 성공한 걸 재사용"하는 최적화와 "부하를 여러 리소스에
  분산"하는 목적은 서로 충돌할 수 있다. 한 번의 논리적 작업(여기서는 챕터 한 편)이 내부적으로
  여러 번의 API 호출로 쪼개진다면, 그 여러 호출 자체를 분산 대상으로 볼지 검토해야 한다.
- 모델(번역 품질/톤에 영향)과 키(단순 인증/쿼터 자원)는 "재시도 시 유지할지" 판단 기준이 다르다 -
  품질에 영향을 주는 축은 유지하고, 순수 자원 축만 분산시키는 식으로 구분해서 접근했다.

  `dowoo-python/tests/test_gemini_client.py`
  (`test_multi_chunk_translation_round_robins_key_after_each_success_to_spread_rpm_load`)
